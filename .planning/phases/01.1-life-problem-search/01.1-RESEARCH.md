# Phase 01.1: Life Problem Search — Research

**Researched:** 2026-03-28
**Domain:** Client-side fuzzy search, Astro static JSON endpoints, Preact search island
**Confidence:** HIGH

---

## Summary

The project already has everything needed for this phase to succeed quickly. All 701 verses across 18 chapters carry `questions` (10,679 natural-language question strings, ~15 per verse across 4 life pillars) and `themes` arrays — both already validated by the Zod schema in `src/content.config.ts`. This means no content backfill is required; the search index is derivable entirely from existing data.

The technical approach is a two-piece architecture: (1) a build-time Astro JSON endpoint at `src/pages/search-index.json.ts` that reads the content collection and emits a compact search index (~822 KB raw, ~243 KB gzipped), and (2) a Preact island (`LifeSearch.tsx`) that lazily fetches the index on first user focus, runs Fuse.js in-memory, and renders a typeahead dropdown and results list. The dedicated search page (`/explore/`) renders the island with `client:load`.

Fuse.js v7.1.0 is the right library for this dataset. At 701 entries with arrays of question strings, fuzzy matching with `ignoreLocation: true` handles natural language variations well. The 243 KB gzipped index is within acceptable payload for a content-focused site (comparable to a single large image). Performance is adequate: Fuse.js on 701 items with ~20 question strings each will search in under 10 ms on modern hardware.

**Primary recommendation:** Build-time JSON endpoint + Fuse.js v7 + Preact island with lazy index fetch on first focus. Do not use Pagefind for this feature — Pagefind indexes HTML pages at build time and cannot search pre-structured JSON question strings with weighted field matching.

### Pattern 1: Build-Time JSON Endpoint

**What:** An Astro API route that reads the content collection at build time and emits a compact JSON search index as a static file.

**When to use:** Always, for search index generation on a static Astro site.

```typescript
// src/pages/search-index.json.ts
import type { APIRoute } from 'astro';
import { getCollection } from 'astro:content';

export const GET: APIRoute = async () => {
  const shlokas = await getCollection('chapters');

  const index = shlokas.map((s) => {
    const d = s.data;
    // Flatten questions from all 4 pillars into one array
    const q: string[] = [
      ...(d.questions.personal_growth ?? []),
      ...(d.questions.career_business ?? []),
      ...(d.questions.health ?? []),
      ...(d.questions.relationships ?? []),
    ];
    return {
      id: d.id,           // "BG-1.1"
      c: d.chapter,       // 1
      v: d.verse,         // 1
      q,                  // ["How do I handle inner conflict...", ...]
      t: d.themes,        // ["duty", "conflict", ...]
      e: d.explanation.slice(0, 200),  // short snippet for result display
    };
  });

  return new Response(JSON.stringify(index), {
    headers: { 'Content-Type': 'application/json' },
  });
};
```

This outputs `dist/search-index.json` at build time. The island fetches it at runtime via `fetch('/search-index.json')`.

**Note on BASE_URL:** GitHub Pages deploys to `/gita-app/` (or a custom domain root). The island must prefix the fetch path with `import.meta.env.BASE_URL`.

### Pattern 2: Preact Island — Lazy Fetch + Fuse.js Search

**What:** Preact component that defers index loading until first user interaction (focus on the search input).

**When to use:** Any search island where the index is large enough that eager loading would hurt LCP.

```tsx
// src/components/islands/LifeSearch.tsx
import { useState, useRef, useEffect } from 'preact/hooks';
import Fuse from 'fuse.js';

interface IndexEntry {
  id: string;   // "BG-1.1"
  c: number;    // chapter
  v: number;    // verse
  q: string[];  // all question strings
  t: string[];  // themes
  e: string;    // explanation snippet
}

const fuseOptions: Fuse.IFuseOptions<IndexEntry> = {
  keys: [
    { name: 'q', weight: 3 },   // questions — highest weight
    { name: 't', weight: 1 },   // themes — lower weight
  ],
  threshold: 0.4,                // tighter than default 0.6 — fewer false positives
  ignoreLocation: true,          // questions can be anywhere in the array
  minMatchCharLength: 3,
  includeScore: true,
  shouldSort: true,
};

export default function LifeSearch({ baseUrl }: { baseUrl: string }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState<Fuse.FuseResult<IndexEntry>[]>([]);
  const fuseRef = useRef<Fuse<IndexEntry> | null>(null);

  const loadIndex = async () => {
    if (fuseRef.current) return;
    const res = await fetch(`${baseUrl}search-index.json`);
    const data: IndexEntry[] = await res.json();
    fuseRef.current = new Fuse(data, fuseOptions);
  };

  const handleSearch = (value: string) => {
    setQuery(value);
    if (!fuseRef.current || value.length < 3) { setResults([]); return; }
    setResults(fuseRef.current.search(value, { limit: 8 }));
  };

  return (
    <div>
      <input
        type="search"
        placeholder="What are you struggling with?"
        onFocus={loadIndex}
        onInput={(e) => handleSearch((e.target as HTMLInputElement).value)}
        value={query}
        // ... Tailwind classes matching project design tokens
      />
      {/* Results list — links to /verse/{chapter}/{verse}/ */}
    </div>
  );
}
```

In the Astro page, pass `baseUrl`:

```astro
<LifeSearch baseUrl={import.meta.env.BASE_URL} client:load />
```


This gives the island instant responsiveness before the index loads.

### Anti-Patterns to Avoid

- **Eager index load on page load:** The index is 822 KB raw. Do not `fetch` it in a `useEffect` on mount — defer to first `onFocus`. On `client:load` the component mounts on page load, but the fetch should only happen on first focus.
- **Pagefind for this feature:** Pagefind is excellent for full-text search of page content but cannot search the structured `questions` field arrays. It is planned for Phase 4 (SEO/full-text) and should not be mixed into this phase.
- **Embedding the index as a prop:** Passing 10,679 questions as Astro props embeds the data in every page's HTML. Always keep it as a separate JSON file fetched on demand.
- **BASE_URL omission:** GitHub Pages uses a base path. `fetch('/search-index.json')` will 404 in production. Always `fetch(`${baseUrl}search-index.json`)`.
- **Fuse.js with default threshold 0.6:** Too permissive for natural-language questions. Use `0.4` with `ignoreLocation: true` for better precision.
- **Re-instantiating Fuse on every keystroke:** Create the Fuse instance once after index load and reuse it. Store in a `useRef`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Fuzzy matching with typo tolerance | Custom Levenshtein distance | Fuse.js | Handles weights, array fields, scoring, sorting — hundreds of edge cases |
| Build-time JSON output | Custom Node.js script | Astro JSON endpoint (`src/pages/*.json.ts`) | Integrates with content collection, runs in Astro's build pipeline, no extra tooling |

**Key insight:** The "life-questions.json" mentioned in the phase description is already effectively embedded in the existing verse JSON files as the `questions` field. There is no separate curation step needed — the search index is a transformation of existing data, not a new content artifact.

## Common Pitfalls

### Pitfall 1: BASE_URL in fetch paths

**What goes wrong:** `fetch('/search-index.json')` returns 404 on GitHub Pages because the site is deployed under `/gita-app/` (or a custom domain with a base path).
**Why it happens:** Developers test on `localhost:4321` where there is no base prefix. The bug only surfaces in production.
**How to avoid:** Always pass `baseUrl={import.meta.env.BASE_URL}` from the Astro page to the Preact island as a prop. Use it in every `fetch()` call inside the island.
**Warning signs:** Search works locally but 404s on GitHub Pages.

### Pitfall 2: Search index not updating after verse data changes

**What goes wrong:** A verse's questions are updated in a chapter JSON file, but the search index at `/search-index.json` still has the old data.
**Why it happens:** The endpoint is built once at build time — this is correct behavior, not a bug. But it's easy to forget.
**How to avoid:** The index is regenerated on every `npm run build`. No caching problem in CI/CD. In local dev, `npm run dev` will recompile the endpoint on file changes automatically.
**Warning signs:** None at runtime — this only matters for local development workflow expectations.

### Pitfall 3: Fuse.js threshold too loose

**What goes wrong:** Queries like "grief" return verses about completely unrelated topics because Fuse matches partial character overlap.
**Why it happens:** Default threshold is `0.6` which allows significant fuzziness. With arrays of 20 questions per entry, some partial match almost always qualifies.
**How to avoid:** Use `threshold: 0.4` and `minMatchCharLength: 3`. Test with queries like "grief", "anger", "I can't sleep" before shipping.
**Warning signs:** Every search returns results regardless of relevance.

### Pitfall 4: ARIA roles missing on combobox pattern

**What goes wrong:** Screen reader users cannot navigate the search suggestions dropdown.
**Why it happens:** Custom combobox components built quickly miss the `role="combobox"`, `aria-autocomplete="list"`, `role="listbox"`, and `role="option"` ARIA attributes.
**How to avoid:** The `input` must have `role="combobox"` and `aria-autocomplete="list"`. The dropdown list must have `role="listbox"`. Each result item must have `role="option"`. Keyboard: ArrowDown/Up navigates options; Enter selects; Escape closes.
**Warning signs:** VoiceOver/NVDA does not announce results.

### Pitfall 5: Large index blocking time-to-interactive

**What goes wrong:** The island fetches 822 KB of JSON on page load, delaying interactivity.
**Why it happens:** Eager fetch in `useEffect` on mount, or `client:load` combined with immediate fetch.
**How to avoid:** Fetch inside `onFocus` handler (lazy). The user must interact with the input before the index loads. With `client:load`, the component hydrates immediately but the index fetch defers.
**Warning signs:** Lighthouse TTI is high on mobile; network waterfall shows the large JSON fetch early.

## Open Questions

1. **Homepage search bar placement**
   - What we know: The homepage (`src/pages/index.astro`) has a hero section with a chapter grid. Adding a search input there would make it the primary entry point.
   - What's unclear: Whether the user wants search only on `/explore/` or also embedded in the hero on the homepage.
   - Recommendation: Build the island as a reusable component; the planner can decide to embed it in both places. Default to `/explore/` only in Phase 01.1; homepage integration can be a follow-up.

2. **Typeahead suggestions vs. full results**
   - What we know: The phase description says "typeahead suggestions." This could mean (a) autocomplete showing matching question strings, or (b) showing verse results directly as you type.
   - What's unclear: UX preference — show the question strings as suggestions, or jump straight to verse cards?
   - Recommendation: Show verse result cards directly (like a search results list), not individual question strings as autocomplete options. The question strings are the search corpus, not the UI display unit. This is clearer and more useful.

3. **How many results to show**
   - What we know: Fuse.js supports a `limit` parameter. Too many results overwhelm; too few miss relevant verses.
   - Recommendation: Default limit of 8 results on the explore page.
